# Narro (텍스트 음성 읽기 앱) 설계서 v0.5

## 0. 문서 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | Narro |
| 플랫폼 | Android Native |
| 주 대상 단말 | Samsung Galaxy S24 Ultra 이상급 |
| 언어/UI | Kotlin, Jetpack Compose |
| 문서 버전 | v0.5 |
| 작성 목적 | 확정 요구사항을 구현 가능한 화면·상태·데이터·오류·검증 기준으로 구체화 |

### 0.1 제품 목표

사용자가 단말의 텍스트 파일을 가져와 화면이 꺼지거나 앱이 백그라운드로 이동한 상태에서도 음성으로 들을 수 있게 한다. 파일과 재생 정보는 단말에 저장하며 별도 서버, 계정, 명시적 클라우드 동기화 기능은 제공하지 않는다.

### 0.2 MVP 범위

- UTF-8, EUC-KR, UTF-16 텍스트 파일 가져오기 및 UTF-8 자동 변환, 목록 표시, 읽기, 삭제
- Android 시스템 TTS를 이용한 음성 재생
- 이전·현재·다음 구간만 메모리에 올리는 대용량 파일 읽기
- 다중 북마크와 읽기 진행률 표시
- 앱 잠금용 4자리 PIN과 선택적 생체인증
- 음성 및 읽기 속도 설정
- Android Auto Backup을 이용한 Google Drive 기기 백업
- 휴대폰의 시스템 언어에 자동으로 맞춰지는 다국어 UI

### 0.3 MVP 제외 범위

- PDF, EPUB, DOCX 등 TXT 이외 형식
- 자체 계정, 서버 저장, 명시적 업로드·동기화 UI
- 외부 TTS API
- 원본 문서 파일 암호화
- PIN 분실 복구
- 비정상 종료 순간의 실시간 재생 위치 복원 및 앱 실행 직후 자동 재생

### 0.4 보안 경계

앱 잠금은 다른 사람이 열린 단말에서 Narro UI에 접근하는 것을 막기 위한 기능이다. 원본 텍스트 파일은 암호화하지 않으므로 루팅, 디버깅 백업, 악성 시스템 권한 등으로 단말 저장소가 노출되는 상황까지 방어하지 않는다.

## 1. 시스템 구조

```text
[Jetpack Compose UI]
        │ 사용자 동작 / 화면 상태
        ▼
[ViewModel + Use Cases]
        ├───────────────┬────────────────┬─────────────────┐
        ▼               ▼                ▼                 ▼
[DocumentRepository] [PlaybackClient] [SettingsRepo] [AppLockRepo]
        │               │                │                 │
        ▼               ▼                ▼                 ▼
 Room + filesDir   Foreground Service  Room/DataStore   Keystore + HMAC
        │               │
        │               ▼
        │        Android TextToSpeech
        │        AudioManager/MediaSession
        ▼
 원본 TXT + 구간 인덱스
```

### 1.1 계층별 책임

| 영역 | 책임 |
|---|---|
| Presentation | 화면 렌더링, 사용자 입력, 진행 상태와 오류 표시 |
| Domain | 파일 가져오기·삭제, 잠금, 북마크, 재생 명령 규칙 |
| Data | Room 트랜잭션, 파일 I/O, 구간 조회, 설정 저장 |
| Playback | TTS 초기화·분할·큐 관리, 오디오 포커스, 백그라운드 재생 |
| Security | PIN 등록·검증·실패 제한, 생체인증 결과 처리 |

### 1.2 권장 기술

| 영역 | 기술 |
|---|---|
| UI | Jetpack Compose, Navigation Compose |
| 적응형 화면 | Material 3 Adaptive `WindowSizeClass`, Window Insets |
| 다국어 | Android Resource Framework, Compose `stringResource`, ICU locale formatting |
| 상태/비동기 | ViewModel, Coroutine, Flow |
| 구조화 데이터 | Room |
| 단순 설정 | Preferences DataStore |
| 파일 선택 | Storage Access Framework |
| 백그라운드 재생 | Foreground Service, MediaSession, MediaStyle 알림 |
| 음성 | `android.speech.tts.TextToSpeech` |
| 생체인증 | AndroidX Biometric |
| 키 관리 | Android Keystore |
| PIN 검증 | Android Keystore HMAC-SHA-256 + 랜덤 salt, 기존 PBKDF2 레코드 자동 마이그레이션 |
| 최소 SDK | API 26 |

### 1.3 다국어 및 지역화 구조

앱 UI 언어는 기본적으로 휴대폰의 시스템 언어 목록과 우선순위를 따른다. 앱에서 별도 언어 값을 저장하거나 임의로 `Locale`을 강제하지 않는다.

```text
휴대폰 언어 목록 변경
  → Android가 가장 잘 일치하는 앱 리소스 선택
  → Compose configuration 변경 감지
  → 화면 재구성
  → 읽기 위치·재생 상태는 유지

지원 언어 없음
  → 기본 영어 리소스로 폴백
```

#### 초기 지원 언어

아래 10개 언어를 초기 정식 지원 범위로 확정한다. 이 표를 Narro의 UI 지원 언어에 대한 단일 기준으로 사용하며, 중국어는 초기 지원 대상에 포함하지 않는다.

| 우선순위 | 언어 | 언어 태그 | Android 문자열 리소스 | 표시 방향 | 역할·대상 |
|---:|---|---|---|---|---|
| 1 | 한국어 | `ko` | `res/values-ko/strings.xml` | LTR | 기준 검수 언어 및 한국 사용자 |
| 2 | 영어 | `en` | `res/values/strings.xml` | LTR | 필수 기본 리소스이자 최종 폴백 |
| 3 | 중남미 스페인어 | `es-419` | `res/values-b+es+419/strings.xml` | LTR | 중남미 스페인어권 공통 |
| 4 | 브라질 포르투갈어 | `pt-BR` | `res/values-pt-rBR/strings.xml` | LTR | 브라질 사용자 |
| 5 | 힌디어 | `hi` | `res/values-hi/strings.xml` | LTR | 인도 힌디어 사용자 |
| 6 | 인도네시아어 | `id` | `res/values-b+id/strings.xml` | LTR | 인도네시아 사용자 |
| 7 | 독일어 | `de` | `res/values-de/strings.xml` | LTR | 독일어권 사용자 |
| 8 | 프랑스어 | `fr` | `res/values-fr/strings.xml` | LTR | 프랑스어권 사용자 |
| 9 | 일본어 | `ja` | `res/values-ja/strings.xml` | LTR | 일본 사용자 |
| 10 | 아랍어 | `ar` | `res/values-ar/strings.xml` | RTL | 아랍어권 사용자 및 RTL 기준 |

- Android `LocaleConfig`에는 `ko`, `en`, `es-419`, `pt-BR`, `hi`, `id`, `de`, `fr`, `ja`, `ar`만 초기 지원 언어로 등록한다.
- 지원하지 않는 시스템 언어는 영어 기본 리소스로 폴백한다.
- 중국어 간체·번체를 포함한 추가 언어는 초기 범위에 포함하지 않는다. 추가 시 별도 우선순위 검토, 번역 검수와 회귀 테스트를 거친다.
- 새 언어 추가가 재생·파일·보안 로직 변경을 요구해서는 안 된다.

#### 구현 규칙

- 모든 사용자 노출 문구는 `strings.xml` 또는 `plurals.xml`에 두며 Kotlin과 Compose 코드에 직접 작성하지 않는다.
- Compose에서는 `stringResource()`와 `pluralStringResource()`를 사용한다.
- 기본 `res/values/strings.xml`에는 앱이 참조하는 모든 문자열이 빠짐없이 있어야 한다.
- 파일명과 사용자가 가져온 본문은 번역하지 않는다.
- 날짜·시간은 시스템 locale과 12/24시간 설정을 따른다.
- 숫자, 퍼센트, 파일 크기 단위는 locale에 맞게 표시하고 내부 저장값은 언어와 무관한 원시 값으로 유지한다.
- 번역문 안에서 단어를 이어 붙이지 않는다. 문장 전체와 서식 인자를 하나의 리소스로 정의해 언어별 어순을 허용한다.
- 파일 개수, 북마크 개수, 남은 문장 수는 복수형 리소스를 사용한다.
- TalkBack 콘텐츠 설명, 오류, 알림 채널명, Foreground Service 재생 알림도 지역화한다.
- UI 이미지에 문자를 포함하지 않는다. 불가피한 경우 언어별 대체 리소스를 제공한다.
- 문자열이 길어져도 잘리지 않도록 버튼은 고정 폭을 피하고 최대 2줄 또는 콘텐츠 기반 너비를 사용한다.
- 아랍어 RTL 지원을 위해 `left/right` 대신 `start/end`를 사용하고 방향성 아이콘만 자동 미러링한다.
- 앱 실행 중 시스템 언어가 바뀌면 화면을 재구성하되 TTS Foreground Service와 현재 읽기 위치는 유지한다.
- Android 13 이상의 시스템 `앱 언어` 설정과 호환되도록 LocaleConfig 자동 생성을 활성화한다. 기본 선택값은 `시스템 기본값`이다.
- 앱 내부 독자 언어 선택기는 MVP에서 제공하지 않는다. 설정의 `앱 언어` 항목은 Android 시스템 앱 언어 설정으로 연결한다.

#### UI 언어와 TTS 언어 분리

앱 버튼과 안내문 언어는 시스템 설정을 따르지만 문서를 읽는 TTS 언어는 문서별 설정이다. 한국어 UI에서 영어 문서를 읽거나 영어 UI에서 한국어 문서를 읽을 수 있어야 한다.

- 새 문서의 TTS 언어 기본값은 시스템 1순위 언어로 설정한다.
- 선택한 음성의 locale과 문서 TTS 언어를 함께 저장한다.
- 해당 언어 음성이 설치되지 않았으면 임의의 다른 언어로 읽지 않고 설치·변경 안내를 표시한다.
- 언어 자동 감지는 보조 제안으로만 사용할 수 있으며 사용자 선택을 덮어쓰지 않는다.

### 1.4 프로젝트 폴더 구조

MVP는 빌드와 의존성 관리를 단순하게 유지하기 위해 단일 `app` 모듈로 시작한다. 아래 구조는 기능 규모가 커졌을 때의 목표 경계다. 현재 MVP 구현은 화면과 화면 상태를 `feature/NarroApp.kt`, `feature/AppViewModel.kt`에 통합하고, TTS도 `playback/service/TtsPlaybackService.kt`에 통합해 작은 코드베이스에서 탐색 비용을 줄였다. 기능이 커질 때 아래 패키지 경계를 따라 분리한다. 기본 패키지는 `com.narro.app`이다.

```text
Narro/
├─ app/
│  ├─ src/
│  │  ├─ main/
│  │  │  ├─ java/com/narro/app/
│  │  │  │  ├─ NarroApplication.kt
│  │  │  │  ├─ MainActivity.kt
│  │  │  │  ├─ navigation/              # 화면 경로, 내비게이션 그래프
│  │  │  │  ├─ feature/
│  │  │  │  │  ├─ lock/                 # PIN·생체인증 화면과 상태
│  │  │  │  │  ├─ documents/            # 문서 목록, 가져오기, 삭제
│  │  │  │  │  ├─ reader/               # 본문 표시, 하이라이트, 재생 제어
│  │  │  │  │  ├─ bookmarks/            # 북마크 목록과 이동
│  │  │  │  │  └─ settings/             # 음성·속도·잠금·앱 언어 설정
│  │  │  │  ├─ domain/
│  │  │  │  │  ├─ model/                # UI·저장 방식에 독립적인 모델
│  │  │  │  │  ├─ repository/           # Repository 인터페이스
```

#### 폴더 운영 규칙

- `feature`는 화면, ViewModel, UI 상태와 이벤트를 기능별로 함께 둔다. 기능 간 화면 구현을 직접 참조하지 않고 `navigation` 또는 Domain 계약을 통해 연결한다.
- `domain`은 Android UI, Room, 파일 API에 의존하지 않는다. Repository 인터페이스와 핵심 업무 규칙만 포함한다.
- `data`는 파일, Room, DataStore, 파서 구현을 담당하며 Domain 인터페이스를 구현한다. UI가 DAO나 파일 구현체를 직접 호출하지 않는다.
- `playback`은 Activity와 독립적으로 동작해야 하며 UI에는 서비스 구현체 대신 명령·상태 인터페이스를 노출한다.
- `security`는 원본 PIN을 저장하거나 로그로 출력하지 않는다. 해시·salt·Keystore 접근을 다른 기능에서 직접 구현하지 않는다.
- 공통 코드는 실제로 둘 이상의 기능에서 사용하는 경우에만 `core`로 이동한다. 기능 전용 코드를 미리 공통화하지 않는다.
- `test`와 `androidTest`는 가능한 한 `main`의 패키지 경계를 동일하게 반영한다. 파서, 재생 상태 전이, PIN 제한 및 Room Migration은 구현과 함께 테스트를 추가한다.
- `message_ko.md`와 `message_en.md`는 문구 검토용 기준 문서다. 빌드에서 사용하는 최종 문자열은 ID를 유지해 각 언어의 `strings.xml`로 반영하며 두 리소스의 ID·서식 인자 일치를 자동 검사한다.
- 원본 문서, 변환 임시 파일, 인덱스 파일은 프로젝트 소스 폴더에 저장하지 않고 런타임의 앱 전용 `filesDir` 하위에서 문서 ID별로 격리한다.

## 2. 핵심 사용자 흐름

### 2.1 앱 진입과 잠금

```text
앱 시작
  │
  ├─ 잠금 꺼짐 ─────────────────────────────▶ 문서 목록
  │
  └─ 잠금 켜짐
       │
       ├─ 생체인증 사용 가능 → 생체인증 요청
       │      ├─ 성공 ──────────────────────▶ 문서 목록
       │      └─ 실패/취소/미지원 ─────────▶ PIN 입력
       │
       └─ 생체인증 미사용 ─────────────────▶ PIN 입력
                  ├─ 성공 ─────────────────▶ 문서 목록
                  └─ 실패 → 실패 횟수 증가/제한 적용
```

최초 설치 시 잠금 기본값은 Off다. 사용자가 설정에서 잠금을 On으로 전환할 때마다 PIN을 두 번 입력해 새로 등록한다. 잠금을 Off로 전환하면 기존 PIN 검증 레코드, 실패 횟수, 제한 시간, PIN 전용 Keystore 키와 생체인증 사용 설정을 초기화한다. 이후 다시 On으로 전환할 때 이전 PIN을 재사용하지 않는다.

### 2.2 파일 가져오기

```text
파일 선택
  → 메타데이터/크기 확인
  → 스트림 샘플 검사(바이너리 여부)
  → 인코딩 판별(UTF-8/EUC-KR/UTF-16)
  → UTF-8 임시 파일로 스트리밍 변환하며 전체 파싱
  → 문장 및 표시 구간 인덱스 생성
  → 원본 파일 확정 이동 + Room 트랜잭션 저장
  → 목록 갱신
```

어느 단계에서든 실패하면 임시 파일과 미완성 DB 데이터를 제거한다. 파싱 중에는 진행률과 취소 버튼을 표시한다.

### 2.3 읽기와 재생

```text
문서 선택
  → 첫 구간 로드
  → 이전/현재/다음 구간만 메모리에 유지
  → 실행
  → Foreground Service가 오디오 포커스 획득
  → 현재 문장을 TTS 최대 입력 길이 이하로 분할
  → utteranceId별 문자 범위 기록
  → 발화 콜백으로 하이라이트/진행률 갱신
  → 현재 문장이 화면 밖으로 이동하면 자동 스크롤
  → 현재 구간 재생과 동시에 다음 구간 및 첫 발화를 사전 준비
  → 구간 경계 도달 시 준비된 다음 구간으로 즉시 전환, 이전 구간 해제
  → 문서 마지막 발화 완료 시 문서 목록으로 이동
```

재생 세션 자체는 프로세스 종료 후 자동 복원하지 않는다. 사용자가 뒤로가기, 다른 문서 선택, 최근 앱 화면에서 Narro 닫기 등 정상 경로로 읽기 화면이나 앱을 종료할 때 현재 위치를 `lastConfirmedPosition`으로 확정 저장하고 내부적으로 재생을 중지한다. 이후 같은 문서를 열면 이 확정 위치부터 이어 읽는다. 충돌, 강제 종료, 시스템의 즉시 프로세스 종료처럼 저장 기회가 없는 비정상 종료에서는 종료 순간 위치를 복원하지 않고 그 전에 저장된 마지막 정상 종료 위치부터 이어 읽는다. 문서를 열었다는 이유만으로 자동 재생하지 않으며 사용자가 재생 버튼을 눌러야 한다.

재생 중이 아니거나 사용자가 일시정지한 상태에서는 본문을 스크롤한 뒤 문장을 더블 탭해 읽기 시작 위치를 변경할 수 있다. 선택한 문장을 즉시 하이라이트하고, 이후 재생 버튼을 누르면 기존 일시정지 위치보다 사용자가 선택한 위치를 우선하여 새 재생 세션을 시작한다. 재생 중, TTS 초기화 중, 일시적 오디오 포커스 상실 상태에서는 위치 선택을 허용하지 않는다.

## 3. 화면 설계

### 3.1 적응형 화면 원칙

주 대상 단말은 Galaxy S24 Ultra지만 6.8인치·3120×1440 물리 해상도나 특정 픽셀 수에 UI를 고정하지 않는다. 실제 앱 창의 dp 너비를 `WindowSizeClass`로 판정해 세로·가로 회전, 글자 크기 변경, 분할 화면에서도 즉시 다시 배치한다.

| 앱 창 너비 | 화면 구성 | 기본 가로 여백 | 본문 최대 너비 |
|---|---|---:|---:|
| Compact, 600dp 미만 | 한 화면에 하나의 주요 콘텐츠, 하단 재생 컨트롤 | 16dp | 사용 가능한 너비 전체 |
| Medium, 600~839dp | 목록·설정의 보조 정보 확장, 읽기 본문 중앙 정렬 | 24dp | 680dp |
| Expanded, 840dp 이상 | 문서 목록+미리보기 등 2-pane 허용, 읽기 본문은 중앙 고정 | 32dp | 720dp |

- Galaxy S24 Ultra 세로 화면은 일반적으로 Compact 레이아웃을 사용하고, 가로 화면이나 넓은 멀티윈도우에서는 현재 앱 창 너비에 따라 Medium 또는 Expanded로 전환한다.
- 화면 크기가 아니라 현재 앱 창 크기를 기준으로 하며 실행 중 너비 등급이 바뀌면 상태를 잃지 않고 즉시 재배치한다.
- 읽기 본문은 넓은 화면에서 무한히 늘리지 않고 최대 720dp로 제한해 한 줄이 지나치게 길어지지 않게 한다.
- 기본 본문 글꼴은 19sp, 줄높이는 30sp로 한다. Android 시스템 글자 크기와 디스플레이 배율을 존중하며 고정 px 글꼴을 사용하지 않는다.
- 텍스트 확대 시 하단 컨트롤이 본문을 가리지 않도록 스크롤 영역에 컨트롤 높이와 시스템 inset만큼 여백을 추가한다.
- 버튼과 아이콘의 최소 터치 영역은 48×48dp로 한다.
- 상태 표시줄, 카메라 홀, 내비게이션 바, 제스처 영역을 `WindowInsets`로 반영하고 콘텐츠가 잘리거나 조작 영역과 겹치지 않게 한다.
- 세로·가로 모드를 모두 지원하며 방향을 Manifest에서 고정하지 않는다.
- 확정 읽기 위치, 현재 발화 하이라이트와 재생 상태는 화면 너비 변경이나 회전 후에도 유지한다. 재생 전 더블 탭으로 지정한 임시 시작 위치는 현재 읽기 화면 세션 동안 유지한다.
- Compact에서는 긴 파일명과 설정 설명을 줄임표 또는 여러 줄로 처리하고 가로 스크롤에 의존하지 않는다.
- Expanded의 2-pane은 목록·북마크·설정에만 적용한다. 읽기 화면은 집중도를 위해 단일 중앙 본문을 유지한다.

```text
Compact (세로/좁은 창)       Medium·Expanded (가로/넓은 창)
┌──────────────────┐        ┌───────────┬────────────────────┐
│ 상단 앱 바       │        │ 문서 목록 │ 선택 문서/보조 정보 │
├──────────────────┤        │           │                    │
│                  │        │           │                    │
│ 읽기 본문        │        └───────────┴────────────────────┘
│                  │
├──────────────────┤        읽기 화면은 너비 720dp 이내로
│ 재생 컨트롤      │        중앙 정렬하고 컨트롤을 고정한다.
└──────────────────┘
```

### S-001 잠금 화면

- 별도 `앱 잠금`, `PIN 입력`, `지문으로 잠금 해제` 설명 문구 없이 Narro 로고, PIN 점 4개, 숫자 키패드만으로 목적을 전달
- Android 잠금 화면과 유사한 크기·간격의 4자리 숫자 키패드 제공
- 네 번째 숫자를 입력하는 즉시 별도 확인 버튼 없이 PIN 검증
- 생체인증이 켜져 있고 사용할 수 있으면 진입 시 한 번 자동 요청
- 생체인증 실패·취소·잠금·미지원 시 PIN으로 전환
- PIN 실패 시 즉시 오류 문구, 거부 진동, PIN 점과 안내 문구의 좌우 흔들림 표시
- 5회 연속 실패 시 30초 잠금
- 실패 횟수와 `lockedUntil`은 앱 재시작으로 우회할 수 없도록 영속 저장
- 성공 시 실패 횟수와 잠금 시간을 초기화

### S-002 문서 목록 화면

- 파일명, 크기, 가져온 시각, 마지막 진행률 표시
- 파일 추가 버튼으로 SAF 파일 선택기 실행
- 가져오는 동안 파일명, 단계, 진행률, 취소 버튼 표시
- 항목 선택 시 읽기 화면으로 이동
- 스와이프 또는 롱프레스 삭제 요청 후 확인 다이얼로그 표시
- 삭제 확인 시 원본 파일, 구간 인덱스, 북마크, 진행률을 hard delete
- 상단 메뉴에서 설정 화면 및 북마크 목록으로 이동

### S-003 읽기 화면

- 상단에 파일명과 뒤로가기 제공
- 본문은 이전·현재·다음 표시 구간만 로드
- 현재 발화 문장 전체를 하이라이트
- 현재 발화 문장이 보이는 화면 영역을 벗어나면 해당 문장이 다시 보이도록 자동 스크롤
- `현재 위치로` 아이콘은 항상 제공하며 누르면 현재 발화 또는 사용자가 선택한 문장으로 이동
- 사용자에게는 아이콘 형태의 재생/일시정지 토글, 이전·다음 구간, 북마크, 현재 위치 이동만 제공하며 별도 중지 버튼은 표시하지 않음
- 재생 버튼 내부의 `재생` 텍스트와 화면의 `1.0배` 속도 표시는 노출하지 않음
- 재생 버튼은 일시정지 상태에서 현재 문장부터 재생하고, 재생 중에는 같은 위치의 일시정지 버튼으로 전환
- 재생 중이 아니거나 사용자 일시정지 상태에서 문장을 더블 탭하면 해당 문장을 선택 위치로 하이라이트하고 다음 재생을 그 문장부터 시작
- 재생 중, 초기화 중, 다른 미디어로 인한 일시정지 상태에서는 문장 더블 탭 위치 변경을 차단
- 뒤로가기, 다른 문서 선택, 앱 종료는 내부적으로 중지 처리하여 위치를 확정 저장하고 TTS 큐와 서비스를 종료
- 문서 마지막 발화가 완료되면 완료 위치와 100% 진행률을 저장하고 문서 목록으로 자동 이동
- TTS 초기화·음성 데이터·재생 오류는 복구 가능한 안내로 표시
- 백그라운드 이동 또는 화면 잠금 시 Foreground Service가 재생 계속

### S-004 북마크 목록 화면

- 파일별 북마크 목록 표시
- 생성 시각과 해당 위치의 짧은 텍스트 미리보기 표시
- 선택 시 문자 위치를 기준으로 해당 구간으로 이동
- 북마크 개별 삭제 지원

### S-005 설정 화면

- 잠금 On/Off
- 잠금 On 시 매번 새 PIN 등록
- 잠금 Off 시 기존 PIN, 실패 제한 상태, PIN 전용 보안 키와 생체인증 설정 초기화
- 생체인증 사용 On/Off
- 설치된 TTS 엔진의 사용 가능한 음성 선택
- 읽기 속도 0.5x~2.0x 선택
- 앱 언어는 `시스템 기본값`으로 표시하고 Android 시스템의 앱 언어 설정 화면으로 연결
- 저장된 음성이 사라진 경우 현재 언어의 기본 음성으로 폴백
- 앱 잠금이 문서 자체를 암호화하지 않는다는 설명 표시

## 4. 기능 명세

### F-001 파일 검증 및 가져오기

| 항목 | 규칙 |
|---|---|
| 형식 | `.txt` 확장자와 MIME은 1차 필터로만 사용하고 내용도 검사 |
| 지원 인코딩 | UTF-8, EUC-KR, UTF-16LE, UTF-16BE |
| 자동 판별 | BOM 확인 → UTF-8 엄격 검증 → EUC-KR 엄격 검증 순서로 판별하고, UTF-16은 BOM을 우선 사용하되 BOM이 없으면 NUL 바이트 배열과 디코딩 유효성을 함께 검사 |
| 내부 저장 | 판별된 입력을 UTF-8로 스트리밍 변환하여 `filesDir/documents/{documentId}.txt`에 저장 |
| 판별 실패 | 어떤 지원 인코딩으로도 유효하게 디코딩되지 않거나 결과가 모호하면 가져오기를 중단하고 지원 인코딩을 안내 |
| 빈 파일 | 공백·개행 제거 후 읽을 문자가 없으면 거부 |
| 바이너리 | NUL 바이트 및 제어문자 비율 등으로 탐지하고 거부 |
| 파일 크기 | 제품 최대 크기를 별도 상수로 정의하고 초과 시 가져오기 전에 거부 |
| 저장 | UTF-8로 정규화한 결과를 `filesDir/documents/{documentId}.txt`에 저장 |
| 중복 | 같은 이름을 허용하고 내부 ID로 구분, 내용 해시가 같으면 중복 경고 |
| 실패 처리 | 임시 파일과 임시 인덱스를 제거하고 DB에는 노출하지 않음 |

ASCII 문자만 있는 파일은 UTF-8과 EUC-KR 양쪽으로 해석할 수 있지만 결과가 같으므로 UTF-8로 판정한다. EUC-KR 판별은 Java/Kotlin의 `CharsetDecoder`를 `CodingErrorAction.REPORT`로 설정해 잘못된 바이트열을 허용하지 않는다. 구현 환경에서 EUC-KR 디코더가 사실상 MS949 확장 문자를 허용하는 경우를 고려해 실제 지원 범위를 테스트로 고정하고, 지원 범위를 UI에 EUC-KR 계열로 표시한다.

UTF-16LE/BE는 BOM이 있으면 이를 제거한 뒤 변환한다. BOM 없는 UTF-16은 짝수 바이트 길이, 예상 위치의 NUL 분포, 엄격 디코딩 성공 여부를 모두 만족할 때만 자동 판정한다. 확신할 수 없는 파일을 임의로 변환해 글자가 손상되는 것보다 명확한 오류로 거부하는 것을 우선한다.

SAF가 제공하는 전체 크기를 신뢰할 수 없는 경우가 있으므로 읽은 원본 바이트 수로 진행률을 보완한다. 전체 크기를 알 수 없으면 퍼센트 대신 처리 바이트와 무한 진행 표시를 사용한다. 변환은 원본 전체를 메모리에 올리지 않고 `InputStreamReader`와 `OutputStreamWriter`의 버퍼 스트림으로 수행한다.

### F-002 스트리밍 파싱 및 구간 인덱스

- 파일 전체를 메모리에 올리지 않고 buffered stream으로 읽는다.
- 구두점과 줄바꿈을 기준으로 문장을 찾되 최대 TTS 입력 길이를 넘는 문장은 안전한 경계에서 추가 분할한다.
- 표시 구간은 일정 문자량 또는 문장 수를 기준으로 생성한다.
- 각 구간에 원문의 시작·끝 바이트/문자 위치와 문장 범위를 기록한다.
- 파싱 규칙 변경에 대비해 `parserVersion`을 저장한다.
- 원본 변경 검출을 위해 SHA-256 `contentHash`를 저장한다.
- 화면은 현재 구간을 기준으로 이전·현재·다음 3개 구간만 유지한다.
- 현재 구간 재생을 시작하는 즉시 다음 구간을 비동기로 사전 로드한다.
- 다음 구간의 텍스트 로드, 문장 범위 확인, 첫 TTS 발화 분할을 현재 구간이 끝나기 전에 완료한다.
- 현재 구간의 마지막 발화 전에 다음 구간의 첫 발화를 동일한 재생 세션의 TTS 큐에 `QUEUE_ADD`로 등록한다.
- 구간 전환 시 파일 I/O, 파싱, 화면 렌더링을 기다리지 않는다. 준비된 메모리 데이터와 TTS 큐를 사용해 연속 재생한다.
- 화면 구간 전환과 하이라이트 갱신은 TTS 재생을 차단하지 않으며, UI 렌더링보다 음성 큐 처리를 우선한다.
- 다음 구간 사전 로드가 완료되기 전에 현재 구간의 마지막 3개 발화에 진입하면 높은 우선순위로 재시도한다.
- 정상 파일과 정상 단말 상태에서는 구간 경계의 파일 로딩으로 인한 추가 재생 지연이 없어야 한다. TTS 엔진 자체가 문장 사이에 삽입하는 고유 간격은 이 기준에서 제외한다.

한글 조합 문자와 surrogate pair가 잘리지 않도록 문자 경계가 아닌 Unicode code point와 문장 경계를 기준으로 분할한다.

### F-003 TTS 재생

| 이벤트 | 동작 |
|---|---|
| 실행 | Foreground Service 시작, 오디오 포커스 획득 후 발화 |
| 사용자 일시정지 | 현재 위치를 유지하고 발화를 멈춤, 재생 세션과 알림은 유지 |
| 사용자 재생 | 일시정지 위치에서 재개, 엔진이 문장 중간 재개를 지원하지 않으면 해당 문장 처음부터 재생 |
| 문장 더블 탭 | 재생 중이 아닐 때 선택한 문장의 인덱스와 시작 문자 위치를 화면 위치와 다음 재생 시작점으로 설정 |
| 화면 꺼짐/단말 잠금 | 계속 재생 |
| 앱 백그라운드 이동 | 계속 재생 |
| 일시적 오디오 포커스 상실 | 재생 일시 중지, 포커스 회복 시 자동 재개 |
| 영구적 오디오 포커스 상실 | 재생 중지, 자동 재개하지 않음 |
| 다른 미디어 종료 | 일시적 상실이었다면 자동 재개 |
| 이어폰·블루투스 출력 분리 | 즉시 중지, 자동 재개하지 않음 |
| 뒤로가기·다른 문서 선택 | 현재 완료 위치를 확정 저장한 뒤 화면 전환 |
| 사용자가 앱 종료 | 가능한 정상 종료 콜백에서 현재 완료 위치를 확정 저장하고 세션 종료 |
| 정상 서비스 종료 | 현재 완료 위치를 확정 저장하고 TTS 큐와 서비스 종료 |
| 비정상 프로세스 종료 | 종료 순간 위치는 저장하지 않으며 기존 확정 위치를 유지 |
| 종료 후 문서 재진입 | 마지막 확정 위치를 로드해 화면을 이동하되 사용자가 실행하기 전에는 발화하지 않음 |
| 문서 마지막 발화 완료 | 완료 위치와 100% 진행률 저장, 재생 서비스 종료, 읽기 화면에서 문서 목록으로 이동 |

- `TextToSpeech.getMaxSpeechInputLength()` 이하로 각 발화를 분할한다.
- 모든 발화에 고유 `utteranceId`를 부여한다.
- 발화 ID와 `documentId`, 구간, 문장, 문자 범위를 매핑한다.
- `UtteranceProgressListener`의 시작·완료·오류 콜백을 단일 재생 상태로 직렬화한다.
- 구간 마지막 발화 뒤에는 사전 준비된 다음 구간의 첫 발화를 `QUEUE_ADD`하여 큐를 끊지 않는다.
- 일시정지는 재생 세션을 유지한다. 뒤로가기·문서 전환·앱 종료 또는 재생 위치 이동 시에만 내부적으로 `stop()`을 호출해 기존 큐를 비운다.
- 알림에서 재생 상태, 파일명, 중지 동작을 제공한다.

### F-004 재생 상태

```text
Idle
 └─ 실행 → Initializing
              ├─ 실패 → Error
              └─ 성공 → Playing

Playing
 ├─ 사용자 일시정지 → PausedByUser
 ├─ 일시적 포커스 상실 → PausedByFocus
 ├─ 뒤로가기/문서 전환/앱 종료/출력 분리 → Stopped
 ├─ 문장 완료 → Playing(next)
 ├─ 문서 완료 → Completed → 문서 목록 이동 → Stopped
 └─ TTS 오류 → Error

PausedByUser
 ├─ 사용자 재생 → Playing
 ├─ 문장 더블 탭 → SelectedPosition → 사용자 재생 → 새 위치에서 Playing
 └─ 뒤로가기/문서 전환/앱 종료 → Stopped

PausedByFocus
 ├─ 포커스 회복 → Playing
 └─ 영구 상실/뒤로가기/문서 전환/앱 종료 → Stopped
```

실행 버튼 연타, 오래된 콜백, 화면 재생성으로 중복 발화가 발생하지 않도록 재생 세션 ID가 현재 세션과 일치하는 콜백만 반영한다.

### F-005 진행률과 북마크

- 재생 중 위치는 메모리의 `currentPlaybackPosition`으로 관리하고, 영속 위치는 `lastConfirmedPosition`으로 구분한다.
- 두 위치 모두 `sentenceIndex`, `characterOffset`, `parserVersion`, `contentHash` 구조를 사용한다.
- 뒤로가기, 다른 문서 선택, 사용자가 앱 종료, 정상적인 서비스 종료 시 마지막으로 완료된 발화의 끝 위치를 `lastConfirmedPosition`으로 원자적으로 저장한다.
- 발화 도중인 미완료 문장은 확정 위치로 저장하지 않아 재진입 시 문장 중간부터 잘려 읽히지 않도록 한다.
- 비정상 종료 시 `currentPlaybackPosition`은 버리고 기존 `lastConfirmedPosition`을 유지한다.
- 문서를 다시 열면 `lastConfirmedPosition`으로 화면과 재생 시작점을 이동한다. 자동 발화는 하지 않고 사용자가 실행 버튼을 누르면 해당 위치부터 읽는다.
- 확정 위치가 없거나 문서를 끝까지 읽은 상태라면 처음부터 시작한다.
- 재생 중 위치가 화면 밖으로 이동하면 현재 발화 문장이 다시 보이도록 자동 스크롤한다. 이미 화면 안에 있으면 불필요하게 스크롤하지 않는다.
- 재생 중이 아닐 때 문장을 더블 탭하면 해당 문장의 `sentenceIndex`와 시작 `characterOffset`을 임시 선택 위치로 사용한다. 재생 버튼을 누르면 선택 위치를 `lastConfirmedPosition`보다 우선한다.
- 문서를 끝까지 읽은 순간에는 첫 구간을 다시 표시하지 않고 문서 목록으로 이동한다.
- 목록의 진행률은 `lastConfirmedPosition`을 기준으로 표시한다.
- 북마크 이동 시 해시와 파서 버전이 같으면 문장 인덱스를 사용한다.
- 해시 또는 파서 버전이 다르면 문자 위치로 재탐색하고 찾지 못하면 북마크가 유효하지 않음을 안내한다.

### F-006 삭제

1. 사용자에게 파일명과 함께 삭제 확인을 요청한다.
2. 재생 중인 문서라면 먼저 재생을 중지한다.
3. Room 트랜잭션에서 관련 구간, 북마크, 진행률을 삭제 대상으로 처리한다.
4. 원본 파일을 삭제한다.
5. 파일 삭제가 성공하면 DB hard delete를 확정한다.
6. 파일 삭제 실패 시 DB 항목을 유지하고 재시도 가능한 오류를 표시한다.

### F-007 PIN과 생체인증

- PIN 원문을 저장하지 않는다.
- 등록 시 `SecureRandom`으로 salt를 생성한다.
- 신규 PIN은 Android Keystore의 반출 불가능 HMAC-SHA-256 키로 salt와 PIN을 서명해 검증값을 만든다. 검증 시 인위적 지연을 두지 않는다.
- 기존 PBKDF2-HMAC-SHA-256 600,000회 레코드는 호환 읽기를 지원하고, 올바른 PIN 인증에 한 번 성공하면 HMAC 방식 레코드로 자동 전환한다.
- Keystore 키로 PIN 검증 레코드를 AES-GCM 암호화해 `noBackupFilesDir`에 저장한다.
- PIN 검증 레코드와 실패 제한 정보는 Google Drive 백업에서 제외한다.
- 생체정보는 앱이 저장하지 않고 OS 인증 결과만 사용한다.
- 잠금을 Off로 전환하면 PIN 검증 레코드, 실패 횟수, 잠금 제한 시간, PIN 전용 HMAC 키와 레코드 암호화 키를 삭제하고 생체인증 설정도 Off로 변경한다.
- 잠금을 다시 On으로 전환하면 이전 PIN 존재 여부와 관계없이 새 PIN 등록과 확인을 요구한다.
- PIN 분실 시 앱 데이터 삭제 후 재설정하도록 안내한다.

### F-008 백업

- 별도 백업·동기화 버튼이나 자체 클라우드 서버는 제공하지 않는다.
- Android Auto Backup을 활성화하여 지원되는 단말에서 Google Drive 기기 백업을 사용한다.
- 원본 문서, Room DB, 일반 설정은 백업 대상이다.
- PIN 검증 레코드, Keystore 종속 데이터, 일시 파일은 백업에서 제외한다.
- 복원 후 PIN 레코드가 없으면 잠금은 Off로 안전하게 초기화한다.
- Android Auto Backup 용량 한도 때문에 모든 대용량 문서의 백업을 보장하지 않으며 UI와 도움말에 이를 명시한다.
- Android 11 이하용 `fullBackupContent`와 Android 12 이상용 `dataExtractionRules`를 모두 정의한다.

```kotlin
    val voiceLocaleTag: String? = null,
    val lockEnabled: Boolean = false,
    val biometricEnabled: Boolean = false
)
```

PIN 검증 레코드는 Room에 저장하지 않는다. 가져오기 중간 상태도 최종 목록 테이블에 넣지 않고 임시 파일과 메모리 작업 상태로 관리한다.

## 6. 오류 처리 기준

| 상황 | 사용자 표시 | 복구 |
|---|---|---|
| 지원하지 않는 인코딩 | UTF-8, EUC-KR, UTF-16만 지원함을 안내 | 지원 형식으로 변환 후 다시 선택 |
| 인코딩 판별 모호 | 자동 판별할 수 없음을 안내 | 파일 인코딩 확인 후 다시 선택 |
| 문자 변환 실패 | 변환된 파일을 저장하지 않고 오류 표시 | 원본 파일 확인 후 재시도 |
| 빈 파일 | 읽을 내용이 없음 | 다른 파일 선택 |
| 바이너리 파일 | 텍스트 파일이 아님 | 다른 파일 선택 |
| 저장 공간 부족 | 필요한 공간 안내 | 공간 확보 후 재시도 |
| 파싱 취소 | 가져오기 취소됨 | 임시 데이터 제거 |
| TTS 초기화 실패 | 음성 엔진 사용 불가 | 엔진 설정 확인/재시도 |
| 음성 데이터 없음 | 언어 데이터 설치 안내 | 시스템 설정 이동 |
| 저장 음성 사라짐 | 기본 음성으로 변경됨 | 설정에서 재선택 |
| 원본 파일 손상 | 문서를 열 수 없음 | 목록에서 삭제/다시 가져오기 |
| 삭제 실패 | 파일 삭제 실패 | 항목 유지 후 재시도 |
| PIN 저장소 손상 | 잠금 정보를 읽을 수 없음 | 앱 데이터 초기화 안내 |

오류는 조용히 무시하지 않는다. 재시도 가능한 오류와 데이터 초기화가 필요한 오류를 구분한다.

## 7. 테스트 계획

### 7.1 파일 가져오기

- 정상 UTF-8, BOM 포함 UTF-8, ASCII 전용 파일
- EUC-KR 한글·영문·숫자·특수문자와 MS949 확장 문자 지원 경계
- BOM 포함 UTF-16LE/BE와 판별 가능한 BOM 없는 UTF-16LE/BE
- 잘못된 바이트열, 잘린 멀티바이트 문자, 홀수 길이 UTF-16, 모호한 인코딩
- 변환 후 원문 내용과 UTF-8 저장 결과가 같은지 검증
- 빈 파일, 공백 전용 파일, NUL 포함 바이너리
- 알려진 크기와 알 수 없는 크기의 SAF URI
- 최대 크기 경계값과 초과 파일
- 복사·파싱 취소, 저장 공간 부족, 읽기 권한 상실
- 같은 이름 파일과 같은 내용 파일
- 실패 후 임시 파일과 DB 잔여물이 없는지 검증

### 7.2 파싱과 구간 로딩

- 구두점, 줄바꿈, 구두점 없는 장문, 이모지와 조합형 문자
- TTS 최대 입력 길이 경계와 초과 문장
- 첫 구간, 중간 구간, 마지막 구간에서 이전·현재·다음 로딩
- 현재 구간 재생 중 다음 구간과 첫 발화가 미리 준비되는지 검증
- 구간 마지막 발화 완료 전에 다음 구간 첫 발화가 TTS 큐에 등록되는지 검증
- 구간 경계에서 추가 파일 I/O나 파싱을 기다리지 않고 연속 재생되는지 통합 테스트
- 느린 저장장치 모의 환경에서도 사전 로드 재시도가 현재 구간 종료 전에 완료되는지 검증
- 빠른 스크롤 중 오래된 로딩 결과 무시
- 파서 버전 및 파일 해시 불일치 시 위치 재탐색

### 7.3 재생

- 일시적 오디오 포커스 상실 후 자동 재개

## 9. 구현 전 확정값

Galaxy S24 Ultra 이상급 단말을 주 성능 기준으로 다음 값을 사용한다. 모든 크기 계산은 원본 파일이 아니라 UTF-8 정규화 결과를 기준으로 한다.

| 항목 | 확정 권장값 | 적용 규칙 |
|---|---:|---|
| 파일 1개 최대 크기 | 20 MiB | 변환 중 20 MiB를 초과하면 즉시 중단하고 임시 파일 삭제 |
| 전체 문서 권장 용량 | 20 MiB | 초과 가져오기는 허용하되 Google Drive 기기 백업이 실패할 수 있음을 경고 |
| 전체 로컬 저장 한도 | 200 MiB | 초과 시 새 파일 가져오기를 막고 삭제 유도 |
| 표시 구간 목표 크기 | 8,192자 또는 40문장 | 먼저 도달하는 기준에서 구간 종료 |
| 표시 구간 최소 크기 | 2,048자 | 자연스러운 문장 경계를 우선하되 지나치게 작은 구간 생성을 방지 |
| 표시 구간 최대 크기 | 12,288자 | 구두점 없는 장문도 이 값을 넘기지 않도록 강제 분할 |
| TTS 발화 목표 크기 | 최대 3,000자 | 엔진의 `getMaxSpeechInputLength()`보다 작은 값 사용 |
| TTS 발화 절대 상한 | `min(3,000, getMaxSpeechInputLength())` | Unicode code point와 문장·공백 경계에서 분할 |
| 다음 구간 사전 로드 시작 | 현재 구간 재생 시작 즉시 | 이전·현재·다음 3개 구간만 메모리에 유지 |
| 사전 로드 긴급 재시도 | 마지막 3개 발화 진입 시 | 다음 구간이 아직 준비되지 않은 경우 우선순위 상향 |
| 신규 PIN 검증 방식 | Keystore HMAC-SHA-256 | 반출 불가능한 256비트 HMAC 키 사용, 인위적 지연 없음 |
| 기존 PIN 호환 방식 | PBKDF2-HMAC-SHA-256 600,000회 | 정상 인증 1회 후 신규 HMAC 방식으로 자동 전환 |
| 신규 PIN 검증 목표 시간 | 100ms 이내 | 네 번째 숫자 입력 직후 사용자가 지연을 느끼지 않게 결과 표시 |
| 신규 PIN 검증 최대 허용 시간 | 250ms | 저전력·발열 상태에서도 입력 피드백을 지연시키지 않음 |
| salt 길이 | 16바이트 | `SecureRandom`으로 PIN 등록마다 새로 생성 |
| PIN 검증값 길이 | 32바이트 | 알고리즘 구분값·salt와 함께 버전 저장 |
| 바이너리 NUL 기준 | 디코딩 후 1개 이상 | UTF-16 판별이 끝난 뒤 본문에 U+0000이 남으면 거부 |
| 제어문자 비율 | 1% 초과 시 거부 | 탭, CR, LF와 일반적인 Unicode 서식 문자는 계산에서 제외 |
| 인코딩 판별 샘플 | 최초 64 KiB | 샘플 판별 후 전체 스트림을 엄격 모드로 다시 검증 |
| 동일 내용 중복 | 경고 후 허용 | 정규화된 UTF-8의 SHA-256 해시로 비교하고 기본 선택은 가져오기 취소 |

### 9.1 값 선정 근거

- 20 MiB 텍스트는 수백만 자에 해당하므로 일반적인 TTS 문서로 충분히 크며 스트리밍 가져오기와 검색 시간을 예측 가능하게 유지한다.
- Android Auto Backup은 앱당 전체 25MB 제한이 있으므로 DB와 설정 여유를 남기기 위해 20 MiB부터 백업 경고를 표시한다. 문서가 여러 개면 총량이 기준이다.
- 8,192자 구간 3개를 유지해도 본문 메모리는 작으며, Compose 렌더링 범위와 다음 구간 사전 준비 시간을 균형 있게 유지한다.
- TTS 엔진별 입력 상한이 다를 수 있으므로 고정 3,000자와 런타임 상한 중 더 작은 값을 사용한다.
- 4자리 PIN 검증은 Keystore 내부 HMAC 키와 앱의 5회 실패·30초 제한을 함께 사용한다. 기존 PBKDF2 레코드는 호환을 위해서만 유지하며 성공 인증 후 즉시 HMAC 방식으로 교체한다.
- 바이너리 검사는 인코딩 판별 전에 NUL만 보고 거부하면 UTF-16 텍스트를 오인하므로 반드시 디코딩 후 수행한다.

### 9.2 성능 수용 기준

Galaxy S24 Ultra 실기기 또는 동급 단말에서 다음 기준을 만족해야 한다.

- 20 MiB 파일 가져오기 중 메모리 피크 증가량 100 MiB 이하
- 구간 이동 후 본문 표시 100ms 이내
- 정상 구간 경계에서 파일 로딩으로 인한 TTS 추가 공백 0ms
- 다음 구간 사전 로드 500ms 이내
